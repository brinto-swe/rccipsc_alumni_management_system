# Alumni Management System Backend Blueprint

## 1. High-Level Architecture
- `config/` contains split settings: `base.py`, `local.py`, and `production.py`.
- `common/` holds reusable primitives: abstract models, pagination, upload helpers, validators, permissions, and task placeholders.
- Each domain app follows a service-selector-permission style so business rules stay out of views.
- Authentication is built on a day-one custom `users.User` model, Djoser registration flows, and JWT login with account-state validation.
- Media handling is Cloudinary-ready through environment-driven storage configuration.
- API documentation is generated by `drf-spectacular` and exposed at `/api/schema/`, `/api/docs/swagger/`, and `/api/docs/redoc/`.

## 4. Settings Split
- `config.settings.base`
  - Installed apps, PostgreSQL config, custom user model, REST framework, JWT, Djoser, drf-spectacular, logging, storage config
- `config.settings.local`
  - Development-first overrides such as default debug behavior and console-friendly email fallback
- `config.settings.production`
  - `DEBUG=False`, SSL redirect, secure cookies, HSTS, and proxy SSL header support

## 5. Auth and User Foundation
- `users.User`
  - Email-first login
  - Role choices: admin, moderator, staff, alumni, guest
  - Account status choices: pending, active, suspended, rejected, deactivated
  - `managed_batch_year` for batch-scoped staff authority
  - `is_verified` for verified-member status
- `accounts`
  - Djoser registration and password flows
  - Custom JWT create endpoint that blocks login for inactive, pending, suspended, rejected, or deactivated accounts
  - Logout endpoint with refresh token blacklist
  - `AccountAuditLog` for auth and admin audit hooks

## 6. Domain Model Summary
- `profiles.AlumniProfile`
  - One-to-one with `users.User`
  - Academic data, contact and privacy settings, location fields, social links, profile picture
- `profiles.AlumniVerificationRequest`
  - Alumni verification workflow reviewed by admin/moderator
- `professions.Profession`
  - Many-to-one from `AlumniProfile`
  - Multiple professional records with date-range validation
- `events.Event`
  - Single extensible event model for Eftar Party, Reunion, Sports Carnival, and Custom Event
- `events.EventSponsor`, `EventRegistration`, `EventGallery`, `EventResult`, `EventReview`
- `posts.Post`, `PostMedia`, `Comment`, `Reaction`, `Report`
- `clubs.Club`, `ClubMembership`
- `teams.Team`, `TeamMember`, `TeamEventRegistration`
- `announcements.AnnouncementCategory`, `Announcement`
- `mentorships.MentorProfile`, `MentorshipRequest`, `MentorshipSession`, `MentorshipFeedback`
- `notifications.Notification`, `NotificationPreference`, `EmailQueue`
- `analytics.AnalyticsSnapshot`

## 7. Model Relationship Explanation
- `User -> AlumniProfile` is `OneToOne`
- `AlumniProfile -> Profession` is `OneToMany`
- `Event -> Sponsors / Registrations / Gallery / Reviews` is `OneToMany`
- `Event -> Result` is `OneToOne`
- `Post -> Media / Comments / Reactions / Reports` is `OneToMany`
- `Club -> ClubMembership` is `OneToMany`
- `Team -> TeamMember / TeamEventRegistration` is `OneToMany`
- `MentorProfile -> MentorshipRequest` is `OneToMany`
- `MentorshipRequest -> MentorshipSession` is `OneToMany`
- `MentorshipSession -> MentorshipFeedback` is `OneToMany`
- `User -> NotificationPreference` is `OneToOne`

## 8. Permission Matrix
| Role | Effective Capabilities |
| --- | --- |
| Admin | Full system access, user approval, role changes, sports carnival creation, analytics, broadcasts, moderation, event override authority |
| Moderator | Notice/news CRUD, club creation, content moderation, report review, broadcasts, analytics read access, verification review |
| Staff | Create Eftar/Reunion/Custom events, manage only own events, create and manage only own-batch teams, upload post-event assets for own events |
| Alumni | Manage own profile and professions, request verification, browse member directory per visibility rules, register for events, post in feed, join clubs, apply as mentor, request mentorship |
| Guest | View public content, browse public directory records, public announcements, public feed, and previous public events |

## 9. Admin Workflow Suggestions
- Approve or reject newly registered users from `/api/admin/users/` or Django admin.
- Review alumni verification requests in `profiles.AlumniVerificationRequest`.
- Review mentor applications and approve from the mentorship module.
- Moderate reported posts/comments from `/api/admin/reports/`.
- Publish broadcasts from `/api/admin/broadcasts/` and inspect queued email from `/api/admin/email-queue/`.
- Monitor `/api/admin/analytics/overview/` for adoption and engagement trends.

## 12. Swagger / OpenAPI Notes

- Extend `extend_schema` examples over time as request/response contracts stabilize.

## 13. Initial Migration Strategy
1. Create a fresh PostgreSQL database for this backend.
2. Copy `.env.example` to `.env`.
3. Set `DB_NAME` to the clean target database, ideally `rccipsc_alumni_system`.
4. Run `python manage.py makemigrations`.
5. Run `python manage.py migrate`.
6. Bootstrap the first admin:
   - `python manage.py bootstrap_system --email admin@example.com --password StrongPassword123`

### Important custom-user note
This project uses a custom user model from the very first migration. Do not point it at an old database that already has `auth/admin` migrations applied from a different user model.

## 14. Seed / Bootstrap Guidance
- `bootstrap_system` creates the first admin safely.
- After bootstrapping, use Django admin or `/api/admin/users/` to create moderator and staff accounts.
- Suggested future seed commands:
  - `seed_reference_data` for announcement categories, club defaults, and demo events
  - `seed_demo_content` for sample posts, teams, and mentor profiles

## 15. Suggested Admin Workflows
- Approve alumni accounts after email activation and manual review.
- Assign `managed_batch_year` when creating staff accounts.
- Review verification requests before enabling `is_verified`.
- Approve mentor applications before public listing.
- Use announcement categories for structure and pin urgent notices.
- Use admin broadcasts instead of direct email blasts for auditability.

## 16. Security Checklist
- Custom user model with role and account-status enforcement
- JWT refresh rotation and blacklist
- Password validators enabled
- Environment-based secrets via `.env`
- File size and extension validation for media/document uploads
- Privacy-aware profile serializers for email, phone, and social links
- Role-based and object-level checks in services and viewsets
- DRF throttling for anonymous, authenticated, auth, and sensitive flows
- Basic logging configured
- Production flags included:
  - `DEBUG=False`
  - secure cookies
  - HSTS
  - SSL redirect
  - CORS/CSRF origin controls
- Audit-friendly records:
  - `AccountAuditLog`
  - `created_by` / `updated_by` fields
  - notification/email queue records

## 18. Celery / Redis Hook Points
- `events.tasks.queue_event_reminder`
- `mentorships.tasks.queue_mentorship_notification`
- `notifications.tasks.dispatch_email_queue`

The current code keeps async side effects behind services so Celery + Redis can be added later without redesigning API layers.

## 19. Supabase Compatibility
- Supabase-hosted PostgreSQL can replace the current database by env-variable changes only.
- Supabase Storage can later sit behind the media layer if Cloudinary is replaced or complemented.
- Supabase Realtime could be added for notification streaming without replacing the existing Django REST architecture.
